### Create DOB: [create-dob/](./create-dob/)

**What is Spore?**
Spore is a DOB (Digital Object) protocol on CKB.

- Unlike Ethereum NFTs where the asset metadata and image are typically stored off-chain (e.g. IPFS), Spore stores the actual content directly inside the cell's `data` field on-chain.
- The content is permanently tied to the cell; it cannot be altered or deleted without burning the cell and reclaiming the CKB. This makes ownership provable and the content truly immutable.

**Spore cell data structure:**

```
data:
  contentType: Bytes   # MIME type string e.g. "image/jpeg"
  content:     Bytes   # raw file bytes stored on-chain
  clusterId:   Bytes   # OPTIONAL — links spore to a collection
type:
  code_hash: SPORE_TYPE_DATA_HASH  # points to Spore contract
  hash_type: "data2"
  args:       SPORE_ID             # unique ID, auto-generated by SDK
lock:
  <user defined>                   # owner's lock script
```

A dApp to mint a Spore DOB (Digital Object) with image content on CKB.

## Key Learnings

### xUDT

- **`KnownScript` is a registry** - the CCC client ships a hardcoded map of well-known contracts (xUDT, Secp256k1, etc.) keyed by name, each storing the `codeHash`, `hashType`, and `cellDeps` needed to use that contract
- **`cellDeps` are mandatory** - validators need to locate and run the xUDT contract to verify a transaction; `addCellDepsOfKnownScripts` looks up the contract's on-chain location from the registry and attaches it
- **Token inputs and CKB inputs are filled separately** - `completeInputsByUdt` finds cells holding the token; `completeInputsByCapacity` then fills CKB for storage costs and fees
- **Token change must be calculated explicitly** - if token inputs exceed the transfer amount, the difference must be sent back to the sender as a new output; it does not happen automatically

### Spore / DOB

- **A Spore cell's `data` field contains** `contentType` (MIME type), `content` (raw file bytes), and optionally `clusterId` (a pointer to a collection the spore belongs to)
- **A cluster is a collection** - create a cluster first, then spores reference it via `clusterId`; without one a spore is a standalone DOB
- **Image bytes are stored directly on-chain** - 1 byte of content = 1 byte of capacity = 1 CKB locked; the raw bytes also travel in the transaction payload, so large images hit the 512KB transaction size limit
